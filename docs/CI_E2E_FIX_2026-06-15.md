# P0: E2E CI Failure Fix (2026-06-15) — FINAL DIAGNOSIS

## TL;DR
E2E workflow fails 13/13 tests because **Supabase secrets don't exist in GitHub repo settings**. Only 2 secrets exist: `CRON_SECRET` and `PRODUCTION_URL`. Need to add `SUPABASE_URL`, `SUPABASE_ANON_KEY`, and `SUPABASE_SERVICE_ROLE_KEY`.

## Definitive diagnosis (verified 2026-06-15)

### Secret list in GitHub
```json
GET /repos/afifghaffarr-source/pricehunt-indonesia/actions/secrets
{
  "total_count": 2,
  "secrets": [
    { "name": "CRON_SECRET" },
    { "name": "PRODUCTION_URL" }
  ]
}
```

**No Supabase secrets exist.** Workflow references `secrets.SUPABASE_URL` etc. which all return empty strings.

### Why CI #208 succeeds but E2E #8 fails
Both workflows reference the same missing secrets. The difference:
- **CI** runs: `lint`, `typecheck`, `vitest` (unit tests with mocked Supabase), `next build`
- **E2E** runs: `next build` (works without env at build time), `next start`, then Playwright hits Supabase at runtime

The build step doesn't error on empty `NEXT_PUBLIC_*` (Next.js just inlines empty strings). But at **runtime**, the Supabase client throws "Missing NEXT_PUBLIC_SUPABASE_URL or NEXT_PUBLIC_SUPABASE_ANON_KEY" → every page renders the error boundary → all 13 tests fail to find their selectors.

## What I tried (commit `e24496b`, pushed)
Added fallback chain `secrets.SUPABASE_URL || secrets.NEXT_PUBLIC_SUPABASE_URL` to support both naming conventions. This was a **defensive fix** but didn't help here because **neither name exists**.

The new "Verify Supabase env vars are set" step now fails fast with a clear error message — making this issue immediately diagnosable in future runs.

## Action required: add 3 secrets

Open: https://github.com/afifghaffarr-source/pricehunt-indonesia/settings/secrets/actions

Click "New repository secret" 3 times, with these names + values (copy from local `.env.local`):

| Secret name | Value source |
|---|---|
| `SUPABASE_URL` | `NEXT_PUBLIC_SUPABASE_URL` in `.env.local` |
| `SUPABASE_ANON_KEY` | `NEXT_PUBLIC_SUPABASE_ANON_KEY` in `.env.local` |
| `SUPABASE_SERVICE_ROLE_KEY` | `SUPABASE_SERVICE_ROLE_KEY` in `.env.local` |

After adding all 3, re-run the workflow (push or workflow_dispatch) and all 19 tests should pass.

## Why I'm not auto-fixing this

I could programmatically create these secrets via the GitHub API using the `secrets/public-key` I already fetched. But:

1. **Security**: repo secrets grant access to production data. Auto-creating them from a chat session crosses a trust boundary.
2. **Confirmation needed**: you may have naming preferences (e.g. `NEXT_PUBLIC_*` vs `SUPABASE_*`).
3. **The fallback chain is in place** so the existing workflow supports either naming convention.

The fix is now just "add 3 secrets with the right names" — 2 minutes of UI work.
